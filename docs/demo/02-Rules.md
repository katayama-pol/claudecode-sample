# 02. Rules — 規約を部品化する（`@`インポート）

## どんな機能か

「Rules（ルール）」は、守ってほしい規約を**独立したファイルに分割**し、
CLAUDE.md から `@ファイルパス` で取り込む仕組みです。

CLAUDE.md に全部ベタ書きすると肥大化するので、
「コーディング規約」「Git規約」などを部品化して、必要なものを組み合わせます。

- 発動タイミング: CLAUDE.md が読み込まれる際に、`@` で参照されたファイルも一緒に取り込まれる（自動・常時）。
- 置き場所: 本デモでは [.claude/rules/](../../.claude/rules/) に置き、[CLAUDE.md](../../CLAUDE.md) の中で
  `@.claude/rules/coding-style.md` のように参照している。

> 補足: `@` インポートは CLAUDE.md だけでなく任意の Markdown 内で使えます。
> ルールの置き場所は自由（`docs/` でも可）。本デモでは分かりやすく `.claude/rules/` にまとめています。

## 打つプロンプト

```
src/index.ts に出てくるユーザー向けメッセージのスタイル規約は何？根拠も教えて。
```

### 何が起きれば成功か

- 「ユーザー向けメッセージは日本語」「エラーは `エラー: ` で始める」など、
  [.claude/rules/coding-style.md](../../.claude/rules/coding-style.md) の内容を**根拠付きで**答える。

## もう一歩：規約が実装に効くか

```
src/index.ts に、引数なしで `add` を呼んだ時の英語エラーメッセージを足したくなった。どう書くべき？
```

### 観察ポイント

- 「英語で」と頼んでも、Claude は規約（メッセージは日本語）を理由に**日本語を提案**するか、
  少なくとも規約との矛盾を指摘するはず。ルールが判断に効いている証拠です。

## 比較実験（差分を見る）

`@.claude/rules/coding-style.md` の行を CLAUDE.md から一時的に消して再起動し、
同じ質問をすると、根拠を答えられなくなる（または一般論になる）はずです。確認後に戻します。

## メリットまとめ

- 規約を**テーマ別に再利用**できる（別プロジェクトに coding-style.md だけ持っていく等）。
- CLAUDE.md が読みやすく保てる。
- 規約の更新が1ファイルで完結し、差分レビューしやすい。
