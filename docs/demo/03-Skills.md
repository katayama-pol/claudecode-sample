# 03. Skills — 「いつものやり方」を手順書にする

## どんな機能か

Skill は、**定型作業の手順書**です。「新しいコマンドを追加するときは、まず既存実装を見て、
ルーティングに足して、ヘルプを更新して、テストして…」という手順を SKILL.md に書いておくと、
Claude がその手順どおりに作業します。
セッション開始時には名前と説明のみが読み込まれ、発動タイミングを満たした時に本文全体が読み込まれます。

- 発動タイミング:
  - **明示的**: `/new-command` のようにスラッシュで呼ぶ。
  - **自動**: SKILL.md の `description` に合う作業だと Claude 自身が判断して呼ぶこともある。
- 置き場所: [.claude/skills/new-command/SKILL.md](../../.claude/skills/new-command/SKILL.md)。
  スラッシュコマンド名は .claude/skills/new-command/ のディレクトリ名から /new-command になる。
  `description` / `when_to_use` が自動発動の判断材料になる。
  `name` は表示名として使われる。

## 打つプロンプト

```
/new-command delete
```

（スラッシュで候補に `new-command` が出ます。引数に追加したいコマンド名 `delete` を渡します）

### 何が起きれば成功か

- いきなり実装を始めず、**SKILL.md の手順に沿って**進む:
  1. 既存の `add`/`list`/`done` を読む → 2. 関数追加 → 3. switch にルーティング追加 →
  4. help 更新 → 5. テスト → 6. `node src/index.ts delete <id>` で動作確認 → 7. 使い方を報告。
- データ操作は store.ts 経由になっている（規約遵守）。

## 自動発動も試す

スラッシュを使わず、自然文でも呼ばれるか確認します。

```
task-cli に、完了済みタスクを一覧から消す clear コマンドを追加して。
```

### 観察ポイント

- `description`（「サブコマンドを追加するとき」）にマッチするため、Claude が**自分から**
  new-command スキルの手順を踏むことがあります（「new-command スキルに従って進めます」等の発言）。

## メリットまとめ

- 作業の**品質と手順が安定**する（指示によるブレ、抜け漏れが減る）。
- ベテランの段取りを**チームに横展開**できる。
- スラッシュコマンドとして**呼び出しやすい**。

> 後片付け: 試しに追加した `delete`/`clear` コマンドが不要なら
> `git checkout src/ ` で元に戻せます。
